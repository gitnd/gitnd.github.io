Vue和React的异同

发表于 2019-08-31

有没有更好的实现呢？

这次总结下目前前端主流框架Vue和React的异同

相同点

  • 都是组件化的写法，具有虚拟dom的概念
  • 都支持通过属性方式(props)，父组件向子组件传参
  • 都支持数据驱动视图，不直接操作dom，更新数据界面自动更新
  • 都支持服务端渲染
  • 都有native方案，支持原生开发，React是React Native,Vue是Weex

不同点

  • 数据绑定方式不同，Vue实现了数据双向绑定，React是单向数据流
  • 组件写法不一样，React是通过jsx的方式，所有html、css全都写进JavaScript中，而Vue是html、css、JavaScript单文件格式，通过webpack、vue、loader解析单文件，相对来说，Vue的语法对初学者更加友好
  • 状态改变方式不同，React状态是不可变的，改变需要通过setState方法更新，而在Vue中，state不是必须的，数据由data属性统一管理
  • 虚拟dom不一样，vue会跟踪每一个组件的依赖关系，不需要重新渲染整个组件树，而对于React来说，每当状态改变时，就会渲染全部组件，需要使用shouldComponentUpdate这个生命周期函数来进行控制
  • 分层结构不同，React只控制view层，是mvc结构，而Vue是mvvm结构

es6 总结(一)

发表于 2019-08-30

多数想法在付诸实践之前都没有意义。

这是一系列有关es6的汇总。

变量声明

在es5中变量的使用没有太多的限制，你可以在任意地方，任何时刻声明赋值变量，甚至变量不用声明也可以直接使用。当然这会造成全局污染问题。
es5的变量使用会遇到的问题有：

  • 变量提升
  • 不声明就可以直接使用
  • 可以重复声明（这和第二条一样，会造成一个问题，覆盖之前的同名变量）

es6定义了新的变量声明方法，let和const,let用来声明变量，const用来声明常量，与es5的不同之处是：

  • 变量必须声明之后才能使用，而且不能重复声明
  • 常量声明后必须赋值，而且不能修改

解构赋值

解构赋值是为了方便的提取数组和对象之中的值。

字符串

字符串提供了许多新的方法，例如：includes、copyWithin…
比较常用的是提供了模板字符串的概念，方便了我们拼接字符串

数组

数组提供了许多新的方法。其中有意思的一个点是，es5和es6中对于占位的不同解读。
es5会将数组中的空占位解释为undefined，es6则会直接跳过，不解释这个占位

键盘侠

每次看微博评论和不同见解的或者说我认为对方这种脑残言论怎么还能苟活于世的人对喷之后都会有一种神清气爽的感觉，虽然有时候会有种压抑郁闷，思考为什么世界上会有这种人，但是键盘治国的奥义就是可以马上的重整旗鼓收拾旧山河，最讨厌的就是那些评论还要设限制，什么必须关注才能回喷的人，我鄙视你呀！

输入url后及闭包

发表于 2019-08-13

复杂的事情讲清楚，你才是真正明白了。

浏览器输入URL之后发生了什么

一共发生了6件事

  1. DNS解析
    • 将域名解析为IP地址
      • 首先读取浏览器缓存
      • 浏览器缓存没有，会读取本地缓存
      • 本地缓存没有，会读取路由缓存
      • 路由缓存没有，会读取运营商缓存
      • 运营商也没有的话，会递归匹配，从com依次，如果没有匹配到，返回报错
  2. tcp握手
    • 三次握手
      • 第一次是浏览器发送请求，通知服务器，让服务器做好准备接收
      • 第二次是服务器发送，通知浏览器已经做好准备，浏览器可以发送
      • 第三次是浏览器再次发送，确保正确连接
  3. 发送请求
    请求报文 协议，请求头，请求体等
  4. 接收响应
    响应报文
  5. 渲染页面
    浏览器调用各种解析器 解析返回的页面
    html 解析为dom树
    css解析为 cssom
    合并生成渲染树
    计算布局 生成页面
    这个步骤会执行多次
  6. 断开连接 tcp四次挥手
    • 浏览器发起，发送给服务器，告诉服务器请求发送完毕，可以关闭
    • 服务器发起，告诉浏览器，请求接收完毕，可以关闭
    • 服务器发起，告诉浏览器，响应发送完毕，可以关闭
    • 浏览器发起，告诉服务器，响应接收完毕，可以关闭

闭包

闭包可以理解为一个对象保存着以键值对存在的值
形成条件：

  1. 函数嵌套
  2. 内部函数可以访问外部函数的局部变量
    优点：
    延长外部函数局部变量的生命周期
    缺点：
    内存泄露，所以在闭包使用完毕之后应该及时清除

事件的防抖和节流、跨域相关

发表于 2019-08-12

保持内心的好奇和平静。

我们总是会遇到事件的频繁触发所导致的请求重复发送及动画掉帧的问题，这些问题有着统一的解决方案，这次做一个总结。

节流

定时器函数通常会遇到的情况是在设置的延迟时间之内回调请求还未执行完毕，或者在鼠标滑动事件中事件的频繁触发，事件节流是为了解决这一个问题所总结出的思想模式，即在一定的延迟时间内事件只执行一次

防抖

防抖多数用来做防止重复提交的问题，比方说按钮频繁点击从而导致的请求重发提交

效果就是只有最后一次触发后的1200ms后事件才会触发执行

跨域

跨域是浏览器的同源策略导致的，浏览器为了数据安全考虑 规定：
只有协议名，域名，端口号完全一致的才可以互相访问数据

解决跨域有很多方法：

  • jsonp
  • corse
  • 设置代理

其中jsonp利用了script天然的没有跨域限制来实现
corse需要后台设置
设置代理可以后台配置也可以前台配置，现在用webpack来设置代理特别简单

行百里者半九十

发表于 2019-08-07

贵在坚持。

今年开始跑步。

应该说，以前也这样开始过，只是都不长久。

春节在家的最后几天，开始沿着长长的人行道慢跑，记得第一次跑了2公里，当时累的
跟条死狗差不多，这让我意识到自己的身体真的太差劲了，而且无才无德，连最基本的
身体素质也是个顶个的差，顿时有股悲凉的气氛浮动在我的四周，于是我决定开始锻炼
身体，算是为了以后的生活做储备。

回到深圳，开始跑步。

第一天我记着多跑了0.5公里，跑完步我把动态更新到朋友圈，算是让朋友做一个见证，也算告诉别人，我这条咸鱼从今天开始不一样了，你们要用发展的眼光来看待我了。

然后我不停的增加着公里数，而且加的分外的猛，我记着不到4天我就加到了10公里，我也不晓得当时是怎么跑下来的，只是当时发到朋友圈的时候心里还有一些小得意，想着马拉松看起来也不是很难的样子嘛？！

乐极生悲的事情发生了，我以前只是听说过跑步之后的后遗症，然后就让我见识到了，当时我记着我就是穿着普通的鞋子跑的，然后第二天脚腿就分外的疼，走路都成问题的那种，百度查一查，哦豁，肌腱炎？

然后，跑步就自然而然的断了，然后就是漫长的休息，第一次的习惯养成在自己的无知下宣告失败。

那时候竟然分外怀念小时候的自己，话说自己从四年级跑到大一，每天学校都会让围着操场跑四圈，当时怎么就没有把脚给跑坏？而且当时自己穿的就是双十几块的回力球鞋。

未解之谜。

然后就买了李宁的跑鞋，等到脚好一点好，又开始慢慢的跑步，然后把公里数稳定在5公里或者多跑一点。

5月中旬随着自己的辞职，跑步当时就突然断了，自己给自己找了很多的借口，什么找工作啥的，一直到7月份，那天晚上忽然思考人生，发现自己到了这个年龄，竟然没有一项拿得出手，说的出去的好习惯，或者说做的算是满意的事情，然后就发泄般的冲出去开始跑步。

然后就是坚持了一周多，台风来了，带来了不休止的雨水，于是跑步又停了。

这次没有长时间的停止，7月底，项目解散，目前赋闲在家，想着把这项习惯培养起来，同时养成别的一些好习惯，于是把博客也捡了起来。

书上说，三个月才可能养成一个好习惯，而一个好习惯坚持一年，俩年，三年，你就会发现你的不一样，我不知道是我换工作的原因还是跑步的原因，忽然发现自己的脾气好了一点，对别人的容忍度也提高了一点。

坚持还是一件分外艰难的事情，毕竟咸鱼是一种更加强大的习惯，每次开始的1.5公里总是分外的艰难，腿脚分外酸疼，但是过了1.5公里，剩下的里程就可以随着惯性跑下来，在今年剩下的日子了，坚持跑步，坚持每日输出，坚持阅读，算是送给自己不断成长的礼物与见证吧。

加油。
